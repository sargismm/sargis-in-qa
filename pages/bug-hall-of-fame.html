#BUG-001
Critical Resolved

Race Condition in Payment Processing Causes Double Charges

E-Commerce Platform Backend / Payments 2023

The Bug

When a user double-clicked the "Pay Now" button faster than the UI could disable it, two simultaneous POST requests were sent to the payment endpoint. The backend processed both before the session lock could prevent it, resulting in duplicate charges and duplicate order creation.

Impact

Financial

~40 customers double-charged before detection. Total exposure: ~$3,200.

Operational

Manual refund processing required. Customer support backlog for 3 days.

Trust

3 customers escalated to credit card disputes, threatening merchant account.

How I Found It

During exploratory testing of the checkout flow, I noticed the "Pay Now" button had no optimistic loading state. I used browser DevTools to throttle the network to "Slow 3G" and manually double-clicked. The network tab showed two successful POST /orders requests with 201 Created responses — both with unique order IDs.

# Reproduced with curl to simulate concurrent requests

$ curl -X POST /api/orders -d '{"cart_id":"abc"}' & \

  curl -X POST /api/orders -d '{"cart_id":"abc"}'

# Both returned 201 Created with different order_ids

# {"order_id": "ORD-9821"} and {"order_id": "ORD-9822"}

Resolution

Frontend: Disabled button immediately on first click via JS flag. Backend: Added idempotency key validation using cart_id + user_id + timestamp_bucket to reject duplicate requests within a 30-second window.

#BUG-002
Security Resolved

Insecure Direct Object Reference Exposes All User Orders

SaaS Dashboard API / Auth 2022

The Bug

The orders API endpoint GET /api/orders/{id} returned order details without verifying the authenticated user owned the requested order. Any logged-in user could enumerate any order by incrementing the integer ID.

Impact

Privacy / GDPR

Full name, address, and order history of all users exposed to any authenticated user.

Compliance

Potential GDPR violation if exploited — mandatory breach notification.

How I Found It

During API testing with Postman, I noticed order IDs were sequential integers. I created two test accounts, placed orders on each, then used Account A's token to request Account B's order ID. The API returned full order data without any authorization error.

Resolution

API layer updated to validate order.user_id === authenticated_user.id before returning data. Order IDs migrated from sequential integers to UUIDs to eliminate enumeration risk.

#BUG-003
High Resolved

Date Timezone Bug Causes Appointments to Book at Wrong Times

Healthcare Scheduling App Frontend / Logic 2023

The Bug

The appointment booking system stored times in UTC but displayed them without timezone conversion. Users in UTC-5 (EST) booking a "3:00 PM" slot were actually booking a 3:00 PM UTC slot — which appeared as 10:00 AM to the provider. Appointments were consistently off by the user's UTC offset.

Impact

Patient Safety

Missed medical appointments. 12 documented cases in a 2-week window.

Revenue

No-show fees reversed. Estimated $4,800 in lost appointment revenue.

How I Found It

During regression testing I checked the feature from a VPN endpoint set to a different timezone (US/Eastern vs UTC). The confirmation email showed "3:00 PM" but the calendar invite attachment contained DTSTART:20230415T150000Z — clearly UTC, not local time.

Resolution

Backend updated to store appointments with explicit timezone metadata. Frontend updated to use the browser's Intl.DateTimeFormat API for display and confirmation emails rebuilt using IANA timezone strings.

#BUG-004
Medium Resolved

CSV Export Silently Truncates Records at 1,000 Rows

Analytics Dashboard Data Export 2022

The Bug

The "Export All" button on a data table silently capped exports at 1,000 rows due to a pagination limit copied from the display layer. No warning was shown. Users with large datasets believed they had complete exports when they had partial data.

How I Found It

Comparing row counts between the UI total counter ("Showing 4,372 results") and the exported CSV line count (1,001 including header). A simple wc -l check revealed the discrepancy.

Resolution

Export endpoint updated to use a cursor-based pagination loop instead of the display pagination. Added a progress indicator and a completion notification showing exact row count exported.
